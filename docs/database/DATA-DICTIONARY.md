## Conventions

| Convention | Meaning |
|---|---|
| All primary keys | `uuid` type — generated by the database (`gen_random_uuid()`) |
| All timestamps | `timestamptz` (timestamp with time zone) — stored in UTC |
| `nullable` | Column may be NULL |
| `NOT NULL` | Column must always have a value |
| `⚠ IMMUTABLE` | This column must never be updated after initial insert |
| `🔒 ROW IMMUTABLE` | Once a specific condition is met, the entire row must not be modified |
| `📋 APPEND-ONLY` | The table must never receive UPDATE or DELETE operations |
| `IDX` | Column has a dedicated index |
| `UQ` | Column has a unique constraint |
| Soft delete | Tables use `is_active` or `is_deleted` flags rather than physical row deletion where noted |

## 1. users

Stores all user accounts — applicants and staff — in a single table. Credential columns are mutually exclusive by auth path.

| Column | Type | Nullable | Constraints | Description |
|---|---|---|---|---|
| `id` | uuid | NOT NULL | PK | Surrogate primary key |
| `email` | varchar(320) | NOT NULL | UQ, IDX | Email address — used for staff login and notifications; populated for applicants from Google profile |
| `google_id` | varchar(255) | nullable | UQ, IDX | Google OAuth subject identifier — populated for applicants only; NULL for staff |
| `password_hash` | text | nullable | — | bcrypt hash — populated for staff only; NULL for applicants |
| `first_name` | varchar(100) | NOT NULL | — | First name |
| `last_name` | varchar(100) | NOT NULL | — | Last name |
| `display_name` | varchar(200) | nullable | — | Optional display name override |
| `is_active` | boolean | NOT NULL | — | `false` = deactivated; user cannot log in; historical records preserved |
| `must_change_password` | boolean | NOT NULL | — | `true` on first staff login; forces password change before any action |
| `last_login_at` | timestamptz | nullable | — | Updated on every successful authentication |
| `created_at` | timestamptz | NOT NULL | ⚠ IMMUTABLE | Row creation timestamp |
| `updated_at` | timestamptz | NOT NULL | — | Last modification timestamp |

**Indexes:** `email` (unique), `google_id` (unique, partial where not null)

**Notes:**
- A user with `google_id` populated is an Applicant; the API must reject staff login attempts via Google for these accounts and vice versa.
- Deactivating a user sets `is_active = false`. Historical actions linked via foreign key remain visible to Admin.
- `password_hash` must never be returned in any API response.

## 2. roles

Lookup table for the 8 defined system roles.

| Column | Type | Nullable | Constraints | Description |
|---|---|---|---|---|
| `id` | uuid | NOT NULL | PK | Surrogate primary key |
| `code` | varchar(50) | NOT NULL | UQ | Role identifier: `APPLICANT`, `ADMIN`, `PROJECT_FOCAL`, `RTEC_MEMBER`, `RTEC_HEAD`, `BUDGET_OFFICER`, `ACCOUNTANT`, `REGIONAL_DIRECTOR` |
| `name` | varchar(100) | NOT NULL | — | Human-readable role name |
| `description` | text | nullable | — | Role description for admin UI |
| `is_active` | boolean | NOT NULL | — | Allows disabling a role without deleting it |

**Notes:**
- Role codes must match the constants used in the workflow engine and permission checks.
- Roles are seeded at deployment and should not be created dynamically by the application.

## 3. user_roles

Many-to-many junction: a user may hold multiple roles simultaneously.

| Column | Type | Nullable | Constraints | Description |
|---|---|---|---|---|
| `id` | uuid | NOT NULL | PK | Surrogate primary key |
| `user_id` | uuid | NOT NULL | FK → users.id, IDX | User receiving the role |
| `role_id` | uuid | NOT NULL | FK → roles.id, IDX | Role being assigned |
| `assigned_at` | timestamptz | NOT NULL | ⚠ IMMUTABLE | When the role was assigned |
| `assigned_by` | uuid | nullable | FK → users.id | Admin who assigned the role; NULL for auto-assigned Applicant role |

**Unique constraint:** `(user_id, role_id)` — a user cannot hold the same role twice.

**Notes:**
- The Applicant role is auto-assigned on first Google login; `assigned_by` is NULL in that case.
- When a role is revoked, the row is deleted (hard delete). The audit log captures the removal event.
- Permission enforcement must check the role the user is *exercising* in the current workflow context, not merely whether they hold any role.

## 4. proposals

Master proposal record. One row per proposal, regardless of how many versions it has.

| Column | Type | Nullable | Constraints | Description |
|---|---|---|---|---|
| `id` | uuid | NOT NULL | PK | Surrogate primary key |
| `applicant_user_id` | uuid | NOT NULL | FK → users.id, IDX | The Applicant who created the proposal |
| `proposal_type_id` | uuid | NOT NULL | FK → proposal_types.id, IDX | The selected proposal type, which determines the form and routing |
| `status` | varchar(60) | NOT NULL | IDX | Current workflow status — one of the 24 defined status codes |
| `title` | varchar(500) | NOT NULL | — | Proposal title as entered by the Applicant |
| `current_version_id` | uuid | nullable | FK → proposal_versions.id | Points to the active version; NULL only briefly during initial creation |
| `created_at` | timestamptz | NOT NULL | ⚠ IMMUTABLE | Row creation timestamp |
| `updated_at` | timestamptz | NOT NULL | — | Updated on every status change or edit |
| `submitted_at` | timestamptz | nullable | — | Timestamp of the first `SUBMITTED_TO_FOCAL` transition |
| `is_locked` | boolean | NOT NULL | — | `true` when status is `APPROVED`, `REJECTED`, or `WITHDRAWN`; prevents further edits |

**Indexes:** `applicant_user_id`, `status`, `proposal_type_id`, `current_version_id`

**Notes:**
- `status` is the single source of truth for workflow position. Changes to status must always be made through the workflow engine, never via direct UPDATE.
- `is_locked = true` must be enforced at the API layer; the database constraint alone is not sufficient.
- Proposals are never physically deleted.

## 5. proposal_versions

🔒 **VERSIONING TABLE — rows are immutable once `is_submitted = true`**

Each submission or resubmission creates a new version row. Submitted versions must never be overwritten.

| Column | Type | Nullable | Constraints | Description |
|---|---|---|---|---|
| `id` | uuid | NOT NULL | PK | Surrogate primary key |
| `proposal_id` | uuid | NOT NULL | FK → proposals.id, IDX | Parent proposal |
| `version_number` | integer | NOT NULL | IDX | Sequential version number starting at 1; unique per proposal |
| `form_template_version_id` | uuid | NOT NULL | FK → form_template_versions.id | The exact form schema used for this version snapshot |
| `created_by` | uuid | NOT NULL | FK → users.id | User who created this version (Applicant or Admin if applicable) |
| `created_at` | timestamptz | NOT NULL | ⚠ IMMUTABLE | Version creation timestamp |
| `submitted_at` | timestamptz | nullable | ⚠ IMMUTABLE after set | Timestamp when the version was formally submitted; NULL while DRAFT |
| `source_version_id` | uuid | nullable | FK → proposal_versions.id | The version this was derived from; NULL for version 1 |
| `status_at_creation` | varchar(60) | NOT NULL | ⚠ IMMUTABLE | Proposal status when this version was created |
| `change_summary` | text | nullable | — | Human-readable summary of changes from the previous version |
| `is_submitted` | boolean | NOT NULL | — | `false` while draft; `true` once formally submitted — **triggers immutability enforcement** |

**Unique constraint:** `(proposal_id, version_number)`

**Immutability rule:** Once `is_submitted = true`, no column in this row or in the associated `proposal_field_values` rows may be updated. The application must enforce this rule. A database trigger may be added to enforce it at the storage layer.

**Indexes:** `proposal_id`, `version_number`, `proposal_id + version_number` (composite)

## 6. proposal_field_values

🔒 **Immutable once the parent `proposal_version.is_submitted = true`**

Stores the complete structured data snapshot for a proposal version — one row per field per version.

| Column | Type | Nullable | Constraints | Description |
|---|---|---|---|---|
| `id` | uuid | NOT NULL | PK | Surrogate primary key |
| `proposal_version_id` | uuid | NOT NULL | FK → proposal_versions.id, IDX | The version this value belongs to |
| `form_field_id` | uuid | NOT NULL | FK → form_fields.id, IDX | The field definition this value maps to |
| `value` | text | nullable | — | The stored field value; serialized as text (JSON for complex types such as table rows) |
| `previous_value` | text | nullable | — | The value from the prior version at the time this version was created; supports field-by-field comparison |
| `changed_by` | uuid | nullable | FK → users.id | Who last modified this field value within a DRAFT version |
| `changed_at` | timestamptz | nullable | — | When this field value was last modified within a DRAFT version |
| `change_reason` | text | nullable | — | Optional reason for change, recorded when an authorized internal user edits a field |

**Unique constraint:** `(proposal_version_id, form_field_id)`

**Indexes:** `proposal_version_id`, `form_field_id`

**Notes:**
- Field values for submitted versions are read-only. The API must reject write attempts on `proposal_field_values` where the parent version has `is_submitted = true`.
- `previous_value` enables version comparison without joining across version rows.
- For repeating-row fields (e.g., budget line items), `value` stores a JSON array.

## 7. comments

💬 **COMMENT TABLE — `visibility` column is the enforcement point for all comment access rules**

| Column | Type | Nullable | Constraints | Description |
|---|---|---|---|---|
| `id` | uuid | NOT NULL | PK | Surrogate primary key |
| `thread_id` | uuid | NOT NULL | FK → comment_threads.id, IDX | The thread this comment belongs to |
| `author_user_id` | uuid | NOT NULL | FK → users.id | Comment author |
| `author_role` | varchar(50) | NOT NULL | ⚠ IMMUTABLE | The role the author was exercising when the comment was created; stored for audit even if role is later removed |
| `body` | text | NOT NULL | — | Comment text content |
| `visibility` | varchar(30) | NOT NULL | IDX | One of: `APPLICANT_VISIBLE`, `FOCAL_AND_INTERNAL`, `RTEC_PRIVATE`, `RTEC_HEAD_ONLY`, `OFFICIAL_WORKFLOW`, `ADMIN_AUDIT_ONLY` |
| `comment_type` | varchar(30) | NOT NULL | — | One of: `OFFICIAL`, `PRIVATE`, `CLARIFICATION`, `FINDING` |
| `status` | varchar(20) | NOT NULL | IDX | One of: `OPEN`, `RESOLVED`, `REOPENED` |
| `parent_comment_id` | uuid | nullable | FK → comments.id | Populated when this is a reply to another comment |
| `resolved_by` | uuid | nullable | FK → users.id | User who resolved the comment |
| `resolved_at` | timestamptz | nullable | — | Timestamp of resolution |
| `created_at` | timestamptz | NOT NULL | ⚠ IMMUTABLE | Creation timestamp |
| `updated_at` | timestamptz | NOT NULL | — | Last update (e.g., if body is edited before submission) |

**Indexes:** `thread_id`, `visibility`, `status`, `author_user_id`

**Visibility enforcement rules (from Roles-and-Permissions §4):**

| Visibility | Who may read |
|---|---|
| `APPLICANT_VISIBLE` | Applicant (own), Project Focal, RTEC Head, Budget Officer, Accountant, RD, Admin |
| `FOCAL_AND_INTERNAL` | Project Focal, RTEC Head, Budget Officer, Accountant, RD, Admin |
| `RTEC_PRIVATE` | The authoring RTEC Member only, RTEC Head |
| `RTEC_HEAD_ONLY` | RTEC Head only |
| `OFFICIAL_WORKFLOW` | Project Focal, Budget Officer, Accountant, RD, Admin |
| `ADMIN_AUDIT_ONLY` | Admin only |

**Critical rule:** Visibility filtering must be enforced server-side in the API. The `visibility` column must never be exposed raw to the Applicant; the API must filter before returning the comment list.

## 9. proposal_workflow_history

📋 **AUDIT TABLE — append-only. No UPDATE or DELETE permitted.**

Records every status transition. This is the authoritative workflow audit trail per proposal.

| Column | Type | Nullable | Constraints | Description |
|---|---|---|---|---|
| `id` | uuid | NOT NULL | PK | Surrogate primary key |
| `proposal_id` | uuid | NOT NULL | FK → proposals.id, IDX | Proposal that transitioned |
| `from_status` | varchar(60) | NOT NULL | ⚠ IMMUTABLE | Status before the transition |
| `to_status` | varchar(60) | NOT NULL | ⚠ IMMUTABLE | Status after the transition |
| `actor_user_id` | uuid | NOT NULL | FK → users.id, IDX | User who triggered the transition |
| `actor_role` | varchar(50) | NOT NULL | ⚠ IMMUTABLE | The role exercised at the time of the action |
| `workflow_action` | varchar(100) | NOT NULL | ⚠ IMMUTABLE | Named action code, e.g. `ENDORSE_TO_RTEC`, `RETURN_TO_APPLICANT` |
| `proposal_version_number` | integer | NOT NULL | ⚠ IMMUTABLE | Version number at the time of the transition |
| `comment` | text | nullable | — | Official comment submitted with the action |
| `session_reference` | varchar(255) | nullable | — | Session ID or request IP where available |
| `transitioned_at` | timestamptz | NOT NULL | ⚠ IMMUTABLE, IDX | UTC timestamp of the transition |

**Indexes:** `proposal_id`, `actor_user_id`, `transitioned_at`

**Immutability rule:** This table is append-only. The application must never issue UPDATE or DELETE statements against it. All entries must include `actor_role` to preserve historical accountability even if a user's role is later changed or removed.

## 10. audit_logs

📋 **AUDIT TABLE — append-only. No UPDATE or DELETE permitted.**

General-purpose audit trail covering all significant system actions beyond workflow transitions: logins, account management, PDF exports, admin content changes, and security events.

| Column | Type | Nullable | Constraints | Description |
|---|---|---|---|---|
| `id` | uuid | NOT NULL | PK | Surrogate primary key |
| `actor_user_id` | uuid | nullable | FK → users.id, IDX | User who performed the action; NULL for automated system actions |
| `actor_role` | varchar(50) | nullable | ⚠ IMMUTABLE | Role being exercised at action time; NULL for system actions |
| `action` | varchar(100) | NOT NULL | IDX | Action code, e.g. `USER_LOGIN`, `USER_DEACTIVATED`, `PDF_EXPORT`, `ROLE_ASSIGNED`, `CONTENT_EDITED` |
| `entity_type` | varchar(100) | NOT NULL | IDX | The entity type affected, e.g. `proposals`, `users`, `comments` |
| `entity_id` | uuid | nullable | IDX | The primary key of the affected row; nullable for non-row actions |
| `before_state` | text | nullable | — | JSON snapshot of the entity before the change; NULL for creation or non-state-change actions |
| `after_state` | text | nullable | — | JSON snapshot of the entity after the change |
| `ip_address` | varchar(45) | nullable | — | IPv4 or IPv6 address from the request; nullable for background jobs |
| `session_reference` | varchar(255) | nullable | — | Session or JWT identifier |
| `created_at` | timestamptz | NOT NULL | ⚠ IMMUTABLE, IDX | UTC timestamp |

**Indexes:** `actor_user_id`, `action`, `entity_type`, `entity_id`, `created_at`

**Immutability rule:** This table is append-only. No UPDATE or DELETE operations are permitted. Application-level guards and, optionally, a PostgreSQL row-level security policy must enforce this.

**Notes:**
- `before_state` and `after_state` must not store sensitive secrets (passwords, tokens). The application layer must scrub such fields before writing.
- Admin content edits to proposals must write a row here with `action = CONTENT_EDITED` and a full JSON diff.

## 11. rtec_reviews

👥 Individual RTEC member review record per proposal version. Freeform remarks only — no numeric scoring.

| Column | Type | Nullable | Constraints | Description |
|---|---|---|---|---|
| `id` | uuid | NOT NULL | PK | Surrogate primary key |
| `proposal_id` | uuid | NOT NULL | FK → proposals.id, IDX | Proposal under review |
| `proposal_version_id` | uuid | NOT NULL | FK → proposal_versions.id | The specific version being reviewed |
| `rtec_group_id` | uuid | NOT NULL | FK → rtec_groups.id | The RTEC group conducting the review |
| `reviewer_user_id` | uuid | NOT NULL | FK → users.id, IDX | The RTEC Member authoring this review |
| `status` | varchar(20) | NOT NULL | — | One of: `DRAFT`, `SUBMITTED` |
| `overall_remarks` | text | nullable | — | Freeform overall remarks from the member |
| `is_submitted` | boolean | NOT NULL | — | `false` while draft; `true` once formally submitted |
| `submitted_at` | timestamptz | nullable | — | Timestamp of final submission; NULL while `is_submitted = false` |
| `created_at` | timestamptz | NOT NULL | ⚠ IMMUTABLE | Record creation timestamp |
| `updated_at` | timestamptz | NOT NULL | — | Last update timestamp |

**Unique constraint:** `(proposal_id, proposal_version_id, reviewer_user_id)` — one review per member per version.

**Indexes:** `proposal_id`, `reviewer_user_id`, `rtec_group_id`

**Notes:**
- Once `is_submitted = true`, the review body must not be modified unless the RTEC Head explicitly reopens it for clarification via a logged workflow action.
- An RTEC member may only view their own review (`reviewer_user_id = current_user`). The RTEC Head may view all reviews for their group.

## 13. Additional Tables Reference

The following tables are included in the ERD and are described briefly here. Full column details follow the same conventions as above.

### applicant_profiles

Extends `users` for Applicant-specific data. One-to-one with `users`. Stores institution, position, contact, and `privacy_consent_given` (boolean, NOT NULL). Privacy consent timestamp is recorded separately as `privacy_consent_at`. The API must not permit any proposal action until `privacy_consent_given = true`.

### staff_profiles

Extends `users` for internal staff data. One-to-one with `users`. Stores `office_id` (FK), `position_title`, and `employee_number`.

### offices / programs

Organizational lookup tables. `programs` (GIA, CEST, SSCP) are linked to `offices`. `proposal_types` link to `programs` to determine routing.

### proposal_types

Catalog of active proposal types. Each type references a `default_form_template_id` and a `program_id` to determine which form loads and which focal receives it.

### proposal_assignments

Records which user is assigned to a proposal in which role. `is_active` flag allows reassignment without deleting history. Admin uses this table when reassigning pending tasks.

### form_templates / form_template_versions / form_sections / form_fields / form_calculations

Form schema registry. `form_template_versions` records schema changes over time with a `schema_version` identifier. `form_fields` stores the full field definition including `input_type`, `validation_rules` (JSON), and `calculation_formula`. `form_calculations` documents Excel-derived formulas separately for Budget Officer review.

### workflow_definitions / workflow_steps / workflow_transitions

Static configuration tables that the workflow engine reads at runtime. `workflow_transitions` encodes the allowed `(from_status, to_status, actor_role, action_code)` tuples. The engine validates each transition request against this table before executing.

### budget_reviews / accounting_reviews

Track the Budget Officer's and Accountant's review state per proposal version. Each endorsement or return action is logged in `proposal_workflow_history`; these tables store the review body and findings.

### rd_decisions

Stores the Regional Director's final decision per proposal version. `decision` is one of `APPROVED`, `DEFERRED`, `REJECTED`, `RETURNED`. Once the proposal reaches a terminal status, this row must not be modified.

### notifications

In-app notification records. One row per recipient per event. `is_read` and `read_at` support unread badge counts. Email notifications are out of scope for the MVP (confirmed by supervisor 2026-07-01).

### email_logs

Future email delivery tracking table. Included in the schema for forward compatibility. Not active in the MVP.

### system_settings

Key-value store for runtime configuration values managed by the Admin. Sensitive values must not be stored here; use environment variables for secrets.

### user_invitations / password_reset_tokens

Short-lived token tables for staff account activation and password reset flows. `expires_at` and `used` columns enforce single-use, time-limited access.

## 14. Immutability and Integrity Summary

| Rule | Affected Tables | Enforcement Layer |
|---|---|---|
| Submitted proposal versions must never be overwritten | `proposal_versions`, `proposal_field_values` | Application API guard + optional DB trigger |
| Workflow history is append-only | `proposal_workflow_history` | Application layer; no UPDATE/DELETE |
| Audit log is append-only | `audit_logs` | Application layer; optional PostgreSQL RLS |
| RTEC member reviews locked after submission | `rtec_reviews` | Application API guard |
| RTEC consolidation locked after submission | `rtec_consolidations` | Application API guard |
| `created_at` columns are set once at insert | All tables | `DEFAULT now()` + no UPDATE of this column |
| `author_role` in comments stored at write time | `comments` | Written at comment creation; not derived from current role |
| `actor_role` in workflow history stored at write time | `proposal_workflow_history`, `audit_logs` | Written at action time; not derived from current role |
